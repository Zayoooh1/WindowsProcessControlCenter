Napraw krytyczny problem w aplikacji Windows Process Control Center.

PRIORYTET 1 — napraw przycisk Browse przy tworzeniu profilu:
Aktualnie po kliknięciu Create profile, wpisaniu nazwy profilu i kliknięciu Browse, nie otwiera się natywne okno eksploratora Windows do wyboru pliku wykonywalnego `.exe`.

Wymagane zachowanie:
- Przycisk Browse ma działać ZAWSZE.
- Browse nie może zależeć od tego, czy pole path jest puste, czy niepuste.
- Browse nie może zależeć od tego, czy nazwa profilu jest już wpisana, czy nie.
- Browse nie może być blokowany przez walidację formularza.
- Kliknięcie Browse ma natychmiast otworzyć natywny picker Windows do wyboru pliku `.exe`.
- Po wybraniu pliku `.exe`, pełna ścieżka do pliku ma zostać wpisana do pola path / executable path w oknie tworzenia profilu.
- Jeżeli użytkownik anuluje wybór pliku, aplikacja nie może się wysypać, nie może czyścić formularza i nie może pokazywać błędu — po prostu zostawia obecny stan bez zmian.
- Picker ma filtrować pliki wykonywalne:
  - `Executable files (*.exe)`
  - opcjonalnie `All files (*.*)`
- Picker powinien startować sensownie, np. w ostatnio używanym folderze, folderze Program Files albo domyślnym folderze systemowym, ale nie jest to najważniejsze.
- Najważniejsze: okno wyboru pliku musi się otwierać bez żadnych problemów.

Sprawdź dokładnie, dlaczego obecnie Browse nie działa:
- Czy event kliknięcia Browse w ogóle dochodzi do backendu / natywnej warstwy aplikacji.
- Czy przycisk nie jest przykryty niewidzialnym elementem UI.
- Czy problem nie wynika z błędnego hit-testingu, złych współrzędnych, złego ID kontrolki albo overlayu.
- Jeżeli UI jest oparte o WebView2, sprawdź most JavaScript ↔ C++ / native bridge.
- Jeżeli UI jest rysowane natywnie Win32, sprawdź obsługę kliknięcia, identyfikatory przycisków i dispatch eventów.
- Sprawdź, czy funkcja otwierająca picker nie jest martwa, niepodpięta albo wywoływana pod złym warunkiem.
- Sprawdź, czy picker nie jest tworzony bez poprawnego owner HWND.
- Sprawdź, czy picker nie jest wywoływany z nieprawidłowego wątku.
- Jeżeli używasz COM/IFileOpenDialog, upewnij się, że COM jest poprawnie inicjalizowany tam, gdzie trzeba.
- Jeżeli obecnie używany jest jakiś fallback albo custom picker, zamień go na stabilny natywny Windows file picker.

Preferowana implementacja:
- Użyj natywnego Windows file picker.
- Najlepiej `IFileOpenDialog` / COM, jeśli projekt już używa Win32/C++.
- Dialog ma być modalny względem głównego okna aplikacji.
- Dialog ma zwracać absolutną ścieżkę do wybranego `.exe`.
- Dodaj bezpieczne logowanie/debug info do konsoli/logów aplikacji:
  - `Browse clicked`
  - `Opening executable picker`
  - `Picker cancelled`
  - `Picker selected: <path>`
  - `Picker failed: <error>`
- Logi mają pomóc potwierdzić, czy kliknięcie w ogóle dochodzi do kodu.

PRIORYTET 2 — lista procesów ma ładować się automatycznie na start:
Aktualnie po uruchomieniu aplikacji lista procesów pokazuje 0 procesów i ładuje się dopiero po kliknięciu Refresh.

Wymagane zachowanie:
- Po uruchomieniu aplikacji lista procesów ma załadować się automatycznie.
- Użytkownik nie powinien musieć klikać Refresh, żeby zobaczyć procesy.
- Przycisk Refresh nadal ma działać ręcznie i odświeżać snapshot procesów.
- Autoload powinien używać tej samej sprawdzonej ścieżki kodu co Refresh, żeby nie było dwóch różnych implementacji.
- Jeżeli procesy są ładowane asynchronicznie, UI powinno zostać zaktualizowane po zakończeniu ładowania.
- Nie blokuj startu aplikacji na długo — ale poprawność jest ważniejsza niż mikro-optymalizacja.
- Po starcie licznik procesów w górnym pasku również ma się aktualizować.

PRIORYTET 3 — sortowanie procesów alfabetycznie:
Lista procesów ma być domyślnie posortowana alfabetycznie po nazwie procesu.
Wymagane zachowanie:
- Sortuj rosnąco po nazwie procesu, case-insensitive.
- Przykład: `chrome.exe`, `Discord.exe`, `explorer.exe` mają być uporządkowane logicznie bez rozróżniania wielkości liter.
- Jeżeli nazwa procesu jest taka sama, jako drugi warunek sortowania użyj PID rosnąco.
- Sortowanie ma działać zarówno przy automatycznym ładowaniu na start, jak i po ręcznym Refresh.
- Nie rozwal istniejącego wyszukiwania/filtrowania.
- Jeżeli użytkownik wpisze coś w search, filtr ma działać na już posortowanej liście albo wynik również ma być posortowany alfabetycznie.

Zakres zmian:
- Nie rób redesignu całej aplikacji.
- Nie zmieniaj layoutu, kolorów, wyglądu tabeli ani sidebaru, chyba że jest to absolutnie konieczne do naprawy kliknięcia Browse.
- Nie usuwaj istniejących funkcji profili.
- Nie zmieniaj formatu zapisu profili, chyba że okaże się to konieczne — wtedy wyjaśnij dlaczego.
- Nie wprowadzaj dużej architektury na siłę.
- Skup się na stabilnej, minimalnej i konkretnej naprawie.

Kolejność pracy:
1. Najpierw znajdź kod odpowiedzialny za okno tworzenia profilu i przycisk Browse.
2. Sprawdź, czy kliknięcie Browse jest poprawnie obsługiwane.
3. Napraw otwieranie natywnego pickera `.exe`.
4. Przetestuj scenariusze:
   - Create profile → pusta nazwa → Browse → picker się otwiera.
   - Create profile → wpisana nazwa → Browse → picker się otwiera.
   - Create profile → path puste → Browse → picker się otwiera.
   - Create profile → path już wpisane → Browse → picker się otwiera i może podmienić path.
   - Anulowanie pickera nie psuje formularza.
   - Wybranie `.exe` wpisuje pełną ścieżkę do pola.
5. Dopiero potem napraw automatyczne ładowanie procesów na starcie.
6. Następnie dodaj sortowanie alfabetyczne procesów.
7. Na końcu zrób build i test.

Acceptance criteria:
- Kliknięcie Browse zawsze otwiera natywne okno wyboru pliku `.exe`.
- Wybrany plik `.exe` pojawia się w polu executable path.
- Anulowanie wyboru nie powoduje błędu.
- Lista procesów ładuje się automatycznie po uruchomieniu aplikacji.
- Lista procesów jest domyślnie posortowana alfabetycznie.
- Refresh nadal działa.
- Nie ma regresji w tworzeniu, imporcie, eksporcie i wyświetlaniu profili.
- Nie ma regresji w widoku Processes.
- Projekt się kompiluje.

Po zakończeniu zwróć raport:
- Jakie pliki zmieniłeś.
- Co było przyczyną problemu z Browse.
- Jak dokładnie naprawiłeś picker.
- Jak rozwiązałeś automatyczne ładowanie procesów na starcie.
- Jak dodałeś sortowanie alfabetyczne.
- Jakie testy ręczne wykonałeś.
- Podaj komendy użyte do builda.
- W sekcji „Build i test” podaj pełną ścieżkę do wygenerowanego pliku `.exe`.
